Points System Implementation Roadmap

Purpose
This roadmap defines how QuizMintAI should migrate the current free-tier quiz waiting model from:

- 3 free quiz uses
- 3 hour cooldown

into a points-based free tier such as:

- 100 free points
- point cost per generation
- timed recharge after points are exhausted

This is an implementation roadmap, not just an idea list.

Recommended Product Model

Default recommendation
- free users start with 100 points
- quiz generation consumes points
- when points reach 0, the user must wait for recharge
- full recharge every 12 hours

Phase 1 status
- completed
- version 1 rules are now locked for implementation

Why 12 hours instead of 1 day
- less punishing for legitimate free users
- still strong enough to protect AI costs
- encourages return usage twice per day at most
- easier acquisition loop than a full 24-hour lockout

Finalized version 1 point costs
- prompt-only quiz generation: 25 points
- file-upload quiz generation: 40 points
- version 1 will not use a more complex sliding scale yet

Recommended first release rule
- keep version 1 simple
- start with one fixed cost for prompt-only quiz generation
- optionally use a second higher fixed cost for file-based generation
- do not start with a highly complicated formula unless needed

Finalized version 1 pricing model
- prompt-only quiz generation: 25 points
- file-upload quiz generation: 40 points
- max free balance: 100 points
- recharge window: 12 hours

This gives free users approximately:
- 4 normal prompt generations per recharge window
- or 2 to 3 heavier generations depending on type

Core Migration Goal

Replace this free-tier mental model
- "I get 3 tries, then I wait 3 hours"

With this model
- "I have a free balance of points, and each generation uses some of it"

This gives you:
- more flexible monetization
- cleaner upgrade messaging
- easier future tuning
- better handling for small vs heavy requests

Phase 3 | Shared Points Logic

Goal
- centralize all point calculations and recharge rules

Phase 3 status
- completed
- shared helper module created in `lib/free-tier-points.ts`
- version 1 constants, recharge helpers, affordability checks, and atomic deduction path are now centralized

Create a shared helper module
- example:
  - lib/free-tier-points.ts

Responsibilities
- determine point cost for a request
- determine whether recharge is due
- refill points when recharge time has passed
- validate whether the user has enough points
- atomically deduct points
- return user-facing status payload

Key functions to create
- getQuizGenerationPointCost(input)
- refreshFreeQuizPointsIfDue(user)
- getAvailableFreeQuizPoints(user)
- canAffordQuizGeneration(user, cost)
- deductFreeQuizPoints(userId, cost)
- getNextQuizRechargeAt(user)

Important rule
- all deductions must be atomic
- do not use fragile read-then-write patterns in separate steps

Done when
- the points system rules live in one reusable module instead of being scattered

- app/api/upload-file/route.ts

New backend behavior
- calculate point cost

- reject generation if points are insufficient

  - nextRechargeAt

Example rejection response

Example success response metadata

Recommended Version 1 Decision

If you want the cleanest first implementation, use:
- freeQuizPoints = 100
- prompt quiz cost = 25
- upload quiz cost = 40
- full recharge every 12 hours

Why this is the best first version
- simple to explain
- simple to implement
- enough flexibility for future tuning
- better than 3 uses / 3 hours

Definition of done

The migration is complete when:
- quiz free-tier access is points-based
- the old 3-use cooldown model is no longer active
- users can clearly see balance and recharge timing
- backend enforcement is atomic and reliable
- pro and premium plans bypass or exceed the free limitation cleanly
